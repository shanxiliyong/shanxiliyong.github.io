Mysql:index

李永

2017-12-22

db

概要

  1. 索引需要存储，也就需要空间，索引实际就是一张表，字段更行会有性能损耗
  2. btree

Innodb

聚集索引
有事务
适合写多，读少

MyISAM

非聚集所以
没有事务
适合读多，写少

建索引原则

推荐

  1. 频繁作为where条件的字段
  2. 关联字段可以建索引，例如外键
  3. 排序字段，例如order by name,group by(先排序，后分组)

不适合

  1. where条件用不到的
  2. 频繁更新的字段
  3. 数据值发布比较均匀的不适合建索引，例如男女，这家
  4. 表达数据可以确定行数，二期数据量很少

其他

##　索引失效

Btree hash

MySql常用30种SQL查询语句优化方法

  1. 应尽量避免在 where 子句中使用!=或<>
  2. where 及 order by 涉及的列上建立索引
  3. where 子句中对字段进行 null 值判断，否则将导致引擎放弃使用索引而进行全表扫描
  4. 尽量避免在 where 子句中使用 or 来连接条件
    如：

可以这样查询：

  1. in 和 not in 也要慎用
    对于连续的数值，能用 between 就不要用 in 了
    应尽量避免在 where 子句中对字段进行表达式操作

  2. 应尽量避免在where子句中对字段进行函数操作

  3. 在使用索引字段作为条件时，如果该索引是复合索引，那么必须使用到该索引中的第一个字段作为条件时才能保证系统使用该索引，否则该索引将不会被使 用，并且应尽可能的让字段顺序与索引顺序相一致

  4. 当索引列有大量数据重复时，SQL查询可能不会去利用索引，如一表中有字段 sex，male、female几乎各一半

  5. 索引并不是越多越好，索引固然可以提高相应的 select 的效率，但同时也降低了 insert 及 update 的效率，因为 insert 或 update 时有可能会重建索引，所以怎样建索引需要慎重考虑，视具体情况而定。

  6. 尽量使用数字型字段，若只含数值信息的字段尽量不要设计为字符型，这会降低查询和连接的性能，并会增加存储开销。这是因为引擎在处理查询和连接时会 逐个比较字符串中每一个字符，而对于数字型而言只需要比较一次就够了
